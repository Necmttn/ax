Turn text is not a blob.

BM25 stays blocks first references become edges dashboard stops regexing

Prototype view of the architectural difference between the current `turn.text` pipeline and a richer model where user messages, tool outputs, file refs, citations, and sections become queryable records.

Current shape

works, but lossy
Provider transcript raw-ish

Claude, Codex, Pi, OpenCode, Cursor each produce nested blocks and metadata.

turn.text big rendered string

Sections, citations, pasted docs, tool output, code, and user intent collapse into one text blob.

text_excerpt BM25 first 500 chars

Fast keyword recall, but misses late mentions and cannot tell which part of the text matched.

Dashboard regex fragile

Rendering tries to rediscover structure using string patterns after structure was already lost.

Suggested shape

analyzable graph
turn raw preserved

Keep `turn.text` as the reconstructed source, but stop treating it as the only source of truth.

turn_block ordered parts

User message, tool result, citation, file ref, code block, plan, error, and context become rows.

edges + projections many uses

References, files, URLs, tools, models, costs, and chunks get their own indexed graph paths.

dashboard renders records not regex

Highlights come from block kind, references, citations, and token spans instead of guessing from text.

How parsing actually works

not magic
Prefer provider-native structure Claude/Codex/Pi/OpenCode transcripts already contain message arrays, tool calls, tool results, roles, timestamps, files, and sometimes citations. We parse those before touching rendered text.
Normalize to block candidates Every native message part becomes a candidate: user text, assistant text, tool call, tool result, code block, URL, file mention, citation, system/developer context.
Use text parser only as fallback If a provider only gives us `turn.text`, we split by stable markers: fenced code, headings, URL lines, file paths, XML-ish tags, tool output prefixes, and quoted transcript wrappers.
Attach confidence and provenance Native blocks get high confidence. Regex-inferred blocks get lower confidence plus the rule name, byte span, and source field. The UI can show uncertain blocks differently.
Derive edges after blocks exist URLs, file paths, model names, tool names, citations, and cost/task keywords become graph edges from the block, not fragile annotations on the whole turn.

Role × atom reality check

sample verified
human user
Mostly prose plus low-level atoms like URLs, paths, headings, code fences, wrappers, and quoted text. Tool calls are not expected here.
text atoms
assistant
Assistant prose plus provider-native tool-call blocks. Claude has `tool_use`; Codex has `function_call`; Pi has `toolCall` blocks.
tool call
tool result
Separate tool-result turns. Claude stores tool results under role=user at provider level, but ax classifies them as `message_kind=tool_result`, not human task text.
output atoms
context
System/developer/AGENTS/instructions are context turns. They contain wrappers, headings, paths, and policy sections, not user task intent.
context atoms

Same atoms, different role meaning

system + user + assistant

The low-level parser can run on every turn role. The atom type is stable, but the interpretation depends on the role and message_kind. A file path in a system message is policy/context; a file path in a user message is usually request scope; a file path in an assistant message is often plan/evidence/progress.

          This is the symbol analyzer layer: detect possible code symbols in
          turn text, classify what kind of symbol they are, then resolve them
          against an AST-backed symbol index. Composto-like tools fit behind
          this layer because they can provide AST/IR extraction and file health
          signals. For ax, the product primitive is symbol grounding: files,
          imports, functions, constants, classes, services, and their edges.
          Then turn text is not only searchable; it can be grounded to actual
          code definitions.

          These planning files are not ordinary markdown. They are agent-produced
          state machines encoded as markdown, YAML frontmatter, tables, XML-ish
          sections, and JSON/YAML config. ax should ingest them as
          artifact_document records, split them into
          artifact_block records, and reuse the same atom/edge
          pipeline as turn text. That gives us searchable project state,
          plan-to-file links, open checklist items, verification evidence, and
          skill instructions, cloud workflow definitions, and Claude Code
          Dynamic Workflow runs. It also gives us a way to compare what the
          agent planned against what the session actually changed, what skill
          instructions were available or invoked, whether a scheduled/webhook
          workflow actually ran as designed, and where a dynamic workflow spent
          its agents, tokens, time, and tool calls.

      typed `turn_block` and `artifact_block` records, then build BM25/graph
      indexes over those blocks.
